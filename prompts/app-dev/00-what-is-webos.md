# 00 · webos 是什么 / 我为啥接入它

## 一句话

webos 是给你的 Web 应用提供 **"桌面运行时"** 的浏览器壳。装上 `@webos/host-sdk`，你的应用就能：

- 以**浮动窗口**形式运行（拖拽 / 缩放 / 最大化 / 最小化 / dock 切换）
- 弹出**统一观感的对话框 / 通知 / 上下文菜单**（不用自己写）
- 跟其他应用**共享用户身份 / token / 主题**
- 被**全局搜索（Cmd+K）直达内部子功能**

## 集成进 webos 你能得到什么

| 没接入 | 接入 webos |
|--------|-----------|
| 应用是浏览器一个标签页 | 是桌面浮动窗口，可拖拽缩放 |
| 弹通知要自己写 toast 组件 | 直接 `Webos.notify({ title, level })` |
| confirm 用浏览器原生（丑） | 用 `Webos.dialog.confirm({...})` 跟桌面观感一致 |
| 鉴权 token 各应用各搞一套 | 共享 `Webos.user.token()`，登录一次全应用通用 |
| 主题切换要自己实现 | 跟随 webos 主题自动切深浅色 |
| 应用之间无法通信 | `Webos.message.send(targetAppId, ...)` |
| 用户记不住你的应用 URL | 桌面图标 / 开始菜单 / 全局搜索都能找到 |

## webos 不是

- **不是 OS / 虚拟机** —— 应用还是跑在浏览器里
- **不是应用商店** —— 不强制经过审核 / 商店上架
- **不是 PWA** —— 应用不需要 service worker / manifest.json (PWA)
- **不是 Electron** —— 不是桌面客户端，是浏览器里的 SPA
- **不是微前端** —— 应用之间是 iframe 物理隔离，不共享代码 / DOM
- **不是 UI 库** —— webos 不给你 Button / Table / Form。这些用 MUI / Ant 等

## 和其他方案对比

| 方案 | 特点 | 跟 webos 比 |
|------|------|-------------|
| PWA | 单应用安装 / 离线 / 推送通知 | webos 是**多应用桌面**，PWA 是单应用安装；可叠加（webos 应用本身可以是 PWA） |
| Electron / Tauri | 真桌面应用 / 系统 API / 离线 | webos 是浏览器内桌面壳，**零下载安装**；适合 SaaS 场景 |
| 微前端（single-spa / qiankun） | 多应用共享一个壳 | 微前端通常**栈耦合**（共享 React / Vue），webos **iframe 物理隔离**栈无关 |
| iframe 嵌入 + 自定义主页 | 简单，但功能少 | webos 是"完整桌面"，含窗口管理 / 全局搜索 / 通知中心 / 主题切换 |
| Puter | 类似定位（开源） | Puter 内置文件系统 / 用户系统 / 后端；webos 不内置任何业务，更轻 |

## 适合 webos 的场景

✅ **企业内部多应用门户**：把 OA、CRM、报表系统、监控、文档等几十个内部 Web 系统统一到一个"桌面"
✅ **SaaS 应用集合**：客户进来就是桌面，左侧应用列表，右侧打开窗口
✅ **数据中台 / 安全审计 / DLP**：多个仪表盘 + 告警面板 + 设置面板，需要"工作台"体验
✅ **第三方 SDK 演示场**：把多个 demo 应用集成到一个桌面给客户体验

## 不太适合的场景

❌ **单个 SPA 应用**：你只有一个应用，没必要塞进桌面壳
❌ **toC 消费类应用**：用户不期望"桌面"体验，期望"原生 App"
❌ **重交互的游戏 / 视频编辑**：iframe 沙箱性能损失对这类应用敏感
❌ **离线场景**：webos 是 web，没网就不能用（PWA 离线另说）
